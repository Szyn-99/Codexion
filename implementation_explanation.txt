OVERVIEW OF THE CODEXION IMPLEMENTATION

This document breaks down the core synchronization mechanisms driving the Codexion simulation, specifically focusing on the branchless lock allocation you highlighted, and the priority queue yielding design. 

---

### 1. The Branchless Mutex Ordering
Lines in question: 
a = &codex->dongles[(ld < rd) * ld + (ld >= rd) * rd];
b = &codex->dongles[(ld < rd) * rd + (ld >= rd) * ld];

1.1. Purpose & Functionality
This is a "branchless" mathematical evaluation designed to determine the minimum and maximum values between 'ld' (left dongle) and 'rd' (right dongle) without using standard `if (ld < rd)` statements. 
- If ld is 3 and rd is 4: `(1) * 3 + (0) * 4` = 3 (Minimum).
- If ld is 4 and rd is 3: `(0) * 4 + (1) * 3` = 3 (Minimum).
This guarantees that pointer `a` always points to the dongle with the LOWER index, and `b` points to the HIGHER index.

1.2. Why is this strictly necessary?
This is a classic solution to the "Dining Philosophers" deadlock scenario. If 5 coders sat in a circle and everyone grabbed their left dongle simultaneously, they would all wait on their right dongle infinitely (Circular Wait Deadlock). 
By ensuring that every coder locks `a` (the lowest index) first, and `b` (highest) second, we break the circular wait. The coder at the end of the table will try to grab Dongle 0 first instead of Dongle 5, disrupting the deadlock cycle.

1.3. Use Cases and Considerations
- Why math instead of standard `if` statements? In the 42 Network `Norminette`, functions are strictly limited to 25 lines. An `if/else` block consumes multiple lines. This boolean arithmetic fits the logic on a single line, preserving vertical space. From a strict performance standpoint, it also prevents "CPU Branch Mispredictions", although for simulations of this scale, the primary motivation is line-economy and deadlock aversion.

### 2. The Condition Variable Wait Mechanism (wait_for_pair & wait_on_cond)

2.1. Purpose & Functionality
Once the desired dongles `a` and `b` are locked, the coder must evaluate if it is allowed to use them. For a dongle to be ready, three conditions must be met:
- It is not actively in use (`!d->in_use`).
- It is not on cooldown (`now >= d->available_at`).
- The current coder is at the very front of the priority queue (`d->queue.waiters[0].id == id`).

If true, the loop breaks, the coder successfully returns to `take_two_dongles`, claims them, and returns.

2.2. The Priority Inversion Safe-Yield
If Dongle A is busy, but Dongle B is not... what does the coder do? 
If a coder simply went to sleep using `pthread_cond_wait()` while holding Dongle B's mutex lock, no other coder could ever interact with Dongle B. If a high-priority (EDF) coder arrived needing Dongle B to prevent an imminent burnout, they would be completely paralyzed. 

To solve this, the implementation does a "Lock Yield":
1. Drop Dongle B (`pthread_mutex_unlock(&b->dongle_mutex)`).
2. Sleep until Dongle A is ready (`wait_on_cond(a)`).
3. Re-acquire Dongle B (`pthread_mutex_lock(&b->dongle_mutex)`).

By yielding the locks of resources we aren't immediately blocked on, we allow the Earliest Deadline First (EDF) and FIFO schedulers to continue accepting and reorganizing the shared wait queues safely.

2.3. Use Cases and Considerations
- Context Switching: Instead of polling (spamming the CPU with checks and `usleep(100)`), the condition variables rely on the OS Kernel to suspend the thread with 0 CPU overhead. 
- Timeouts: The timeout interval inside `wait_on_cond` ensures that if a coder is simply waiting out a hardware cooldown, they wake up precisely when the cooldown ends. It also provides a 5ms "heartbeat" to ensure that if the global `finished_simulation` flag halts the program, the sleep aborts smoothly and cleanly without zombie threads remaining stuck.

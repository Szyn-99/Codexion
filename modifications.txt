================================================================================
                    CODEXION - EDF BURNOUT FIX MODIFICATIONS
================================================================================


================================================================================
MODIFICATION 1: Odd/Even Dongle Acquisition Order
================================================================================
File: codexion_croutine.c (function: take_two_dongles)

ISSUE:
  The original code always picked up the lower-numbered dongle first. This
  created a chain of blocking: coder 1 holds dongle 0 while waiting for
  dongle 1, but dongle 1 is held by coder 2 who waits for dongle 2, held
  by coder 3, etc. As a result, only ONE coder could compile per round
  instead of N/2 coders compiling in parallel.

WHY THIS FIXES IT:
  By making odd-ID coders pick up their RIGHT dongle first and even-ID
  coders pick up their LEFT dongle first, adjacent coders reach for
  different shared dongles. This allows non-adjacent coders (e.g. coders
  1, 3, 5) to grab both their dongles simultaneously, maximizing
  parallelism.

BEFORE:
  void take_two_dongles(t_codexion *codex, t_coder *coder, int rd, int ld)
  {
      if (ld < rd)
      {
          take_dongle(codex, coder, ld);
          take_dongle(codex, coder, rd);
      }
      else
      {
          take_dongle(codex, coder, rd);
          take_dongle(codex, coder, ld);
      }
  }

AFTER:
  void take_two_dongles(t_codexion *codex, t_coder *coder, int rd, int ld)
  {
      if (coder->id % 2 == 1)
      {
          take_dongle(codex, coder, rd);
          take_dongle(codex, coder, ld);
      }
      else
      {
          take_dongle(codex, coder, ld);
          take_dongle(codex, coder, rd);
      }
  }

EXAMPLE:
  With 4 coders and the OLD code:
    Time 0:   Coder 3 takes dongles 2,3 -> compiles (only one!)
    Time 210: Coder 2 takes dongles 1,2 -> compiles (serial)
    Time 420: Coder 1 tries dongle 1 -> BURNED OUT (waited 420ms > 410ms)

  With 4 coders and the NEW code:
    Time 0:   Coder 1 takes dongles 0,1 -> compiles }  TWO coders
    Time 0:   Coder 3 takes dongles 2,3 -> compiles }  in parallel!
    Time 210: Coder 2 takes dongles 1,2 -> compiles }  TWO coders
    Time 210: Coder 4 takes dongles 3,0 -> compiles }  in parallel!
    -> No burnout.


================================================================================
MODIFICATION 2: Even Coder Stagger
================================================================================
File: codexion_croutine.c (function: coders_routine)

ISSUE:
  Even with odd/even dongle ordering, which coder wins the shared dongle
  is a thread race. If an even coder wins, it also grabs the neighboring
  odd coder's second dongle, breaking the parallel grouping. With 200
  threads, thread creation takes several milliseconds, making these races
  unpredictable.

WHY THIS FIXES IT:
  A 1ms delay for even coders ensures all odd coders lock their shared
  dongles before any even coder tries. This costs less than 2% of the
  minimum 60ms burnout budget. After round 1, the natural phase offset
  (debug + refactor) keeps the groups separated, and EDF priority handles
  any remaining contention.

BEFORE:
  ld = coder->id - 1;
  rd = coder->id % coder->sim->parse->number_of_coders;
  while (!finished_simulation(coder->sim))

AFTER:
  ld = coder->id - 1;
  rd = coder->id % coder->sim->parse->number_of_coders;
  if (coder->id % 2 == 0)
      usleep(1000);
  while (!finished_simulation(coder->sim))

EXAMPLE:
  With 10 coders and NO stagger:
    Time 0: Coders 1, 4, 7 compile (only 3 out of 5 possible)
            Coder 4 (even) won the race and stole coder 5's dongle
    -> Suboptimal grouping -> some coders burn out

  With 10 coders and stagger:
    Time 0: Coders 1, 3, 5, 7, 9 compile (all 5 odd coders!)
    Time ~210: Coders 2, 4, 6, 8, 10 compile (all 5 even coders!)
    -> Perfect parallel grouping -> no burnout


================================================================================
MODIFICATION 3: Release Dongles on Early Exit
================================================================================
File: codexion_croutine.c (function: coders_routine)

ISSUE:
  When the simulation ends (burnout detected or all compiles done), a
  coder that just acquired two dongles would check finished_simulation()
  and return NULL immediately -- WITHOUT releasing the dongles it was
  holding. Other coders waiting on those dongles would hang forever.

WHY THIS FIXES IT:
  Before returning, both dongles are explicitly released. This unblocks
  any coder still waiting on a dongle's condition variable.

BEFORE:
  take_two_dongles(coder->sim, coder, rd, ld);
  if (finished_simulation(coder->sim))
      return (NULL);

AFTER:
  take_two_dongles(coder->sim, coder, rd, ld);
  if (finished_simulation(coder->sim))
  {
      put_dongle(coder->sim, ld);
      put_dongle(coder->sim, rd);
      return (NULL);
  }

EXAMPLE:
  With the OLD code:
    Coder 1 burns out. Monitor sets sim_over = 1.
    Coder 3 just took both dongles. Checks sim_over -> true.
    Returns without releasing. Coder 2 is stuck on pthread_cond_wait
    forever -> program hangs.

  With the NEW code:
    Coder 3 releases both dongles before returning.
    Coder 2 wakes up, checks sim_over, exits cleanly.
    Program terminates normally.


================================================================================
MODIFICATION 4: Fix Burnout Detection Timing
================================================================================
File: codexion_monitor.c (function: detect_lazy_coder)

ISSUE:
  Two problems:
  (a) The comparison used <= (less-or-equal). This meant a coder whose
      burnout deadline is exactly NOW was flagged as burned out. With
      tight timings, a coder compiling at exactly the deadline would
      be falsely detected.
  (b) The current time was sampled ONCE before the loop, then used to
      check ALL coders. By the time the last coder is checked, the
      timestamp is stale by several milliseconds, causing false positives.

WHY THIS FIXES IT:
  (a) Changed to < (strict less-than): a coder only burns out when the
      deadline has PASSED, not when it's exactly reached.
  (b) Time is re-sampled inside the loop for each coder, so each check
      uses a fresh, accurate timestamp.

BEFORE:
  criteria = get_time_ms();
  i = 0;
  while (i < codex->parse->number_of_coders)
  {
      pthread_mutex_lock(&codex->sim_mutex);
      last = codex->coders[i].last_compile_start;
      pthread_mutex_unlock(&codex->sim_mutex);
      if (last + codex->parse->time_to_burnout <= criteria)
          return (codex->coders[i].id);
      i++;
  }

AFTER:
  i = 0;
  while (i < codex->parse->number_of_coders)
  {
      criteria = get_time_ms();
      pthread_mutex_lock(&codex->sim_mutex);
      last = codex->coders[i].last_compile_start;
      pthread_mutex_unlock(&codex->sim_mutex);
      if (last + codex->parse->time_to_burnout < criteria)
          return (codex->coders[i].id);
      i++;
  }

EXAMPLE:
  Coder 1 last compiled at time 0ms. Burnout = 410ms.
  Deadline = 0 + 410 = 410ms.

  With OLD code (<=):
    At time 410ms: 410 <= 410 -> BURNED OUT (false positive, coder is
    about to compile)

  With NEW code (<):
    At time 410ms: 410 < 410 -> false -> NOT burned out
    At time 411ms: 410 < 411 -> true -> burned out (correct)


================================================================================
MODIFICATION 5: Suppress Log Messages After Simulation Ends
================================================================================
File: codexion_utils.c (function: coder_logs)

ISSUE:
  When a coder burns out, the monitor sets sim_over = 1 and prints
  "burned out". But other coders that were waiting in take_dongle()
  also wake up and print stale messages like "has taken a dongle"
  AFTER the burnout line. The subject requires no messages to appear
  after a burnout.

WHY THIS FIXES IT:
  Before printing, the function checks if sim_over is set. If it is,
  the message is suppressed -- EXCEPT for the "burned out" message
  itself (detected by checking if the log starts with 'b').

BEFORE:
  pthread_mutex_lock(&codex->log_mutex);
  printf("%ld %d %s\n", elapsed, id, log);
  pthread_mutex_unlock(&codex->log_mutex);

AFTER:
  pthread_mutex_lock(&codex->log_mutex);
  if (!codex->sim_over || log[0] == 'b')
      printf("%ld %d %s\n", elapsed, id, log);
  pthread_mutex_unlock(&codex->log_mutex);

EXAMPLE:
  With OLD code:
    410 1 burned out
    410 3 has taken a dongle    <- WRONG: message after burnout
    410 2 has taken a dongle    <- WRONG: message after burnout

  With NEW code:
    410 1 burned out
    (no more messages)


================================================================================
MODIFICATION 6: Allow Zero as Valid Input in ft_atoi
================================================================================
File: parsing_arguments.c (function: ft_atoi)

ISSUE:
  The function treated result == 0 as an error and returned -42. This
  meant passing "0" as the dongle cooldown argument was rejected with
  "Error: Invalid Input", even though 0 is a perfectly valid cooldown.

WHY THIS FIXES IT:
  Instead of checking if the result is zero, it tracks whether at least
  one digit was found. An empty string or all-whitespace string is still
  rejected, but "0" is now accepted.

BEFORE:
  int  ft_atoi(char *str)
  {
      int     i;
      int     space_length;
      long    result;

      i = 0;
      space_length = 0;
      result = 0;
      while (str[i] != '\0')
      {
          if (yes_space(str[i]))
              space_length++;
          else if (((result * 10) + (str[i] - '0')) > 2147483647)
              return (-42);
          else if (yes_int(str[i]))
              result = (result * 10) + (str[i] - '0');
          else
              return (-42);
          i++;
      }
      if (result == 0)
          return -42;
      return (result);
  }

AFTER:
  int  ft_atoi(char *str)
  {
      int     i;
      int     has_digit;
      long    result;

      i = 0;
      has_digit = 0;
      result = 0;
      while (str[i] != '\0')
      {
          if (yes_space(str[i]))
              ;
          else if (yes_int(str[i]))
          {
              if (((result * 10) + (str[i] - '0')) > 2147483647)
                  return (-42);
              result = (result * 10) + (str[i] - '0');
              has_digit = 1;
          }
          else
              return (-42);
          i++;
      }
      if (!has_digit)
          return (-42);
      return (result);
  }

EXAMPLE:
  With OLD code:
    ./codexion 4 410 200 100 100 5 0 edf
    -> "Error: Invalid Input"
    -> "Error: Failed to initialize the simulation"

  With NEW code:
    ./codexion 4 410 200 100 100 5 0 edf
    -> Simulation runs normally with cooldown = 0
